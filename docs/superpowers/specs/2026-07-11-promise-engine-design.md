# The Promise Engine — Design

**Date:** 2026-07-11
**Status:** Approved, ready for implementation planning
**Source idea:** [`PROMISE-ENGINE.md`](../../../PROMISE-ENGINE.md)

An agent that decides the delivery date a marketplace shows at checkout — one it can actually
keep. Built on CRAFT MCP over the Olist (`brazilian-e-commerce-5f7bc95c`) dataset.

---

## 1. Purpose

Olist's delivery promise is distance-aware but blind to **variance** (which lane) and **load**
(which month). The engine sets a promise that hits a 95% on-time target, attributes the number to
its causes, and decides per lane whether to **pad the promise** or **fix the lane**.

Verified ground truth driving the design (already confirmed against live data):

| Lane | Orders | Promised | Median | p95 | Late rate | Gap |
| --- | --- | --- | --- | --- | --- | --- |
| SP | 40,494 | 19.8d | 7d | 20.0d | 5.9% | +0.2 |
| RJ | 12,350 | 27.0d | 12d | 38.0d | 13.5% | **+11.0** |
| CE | 1,279 | 32.0d | 18d | 45.0d | 15.3% | **+13.0** |

Rio is the thesis: **median 12, p95 38.** Not slow — unpredictable.

### Success criteria

1. Given a seller and a destination state, the engine returns a promise, a confidence level, an
   attribution (handling vs. transit), and a pad/fix verdict.
2. Every number is reproducible and derived in Python, never generated by an LLM.
3. The repo runs end-to-end with **zero credentials**.
4. The agent falsifies its own hypotheses before recommending.

### Non-goals

- **Any claim about conversion.** Olist has no clickstream. Conversion is unmeasurable here.
  Claim only: service level, broken promises, and the 1-star reviews they cause.
- Voice, auth, multi-tenancy, real-time carrier feeds.

## 2. Architecture

Four layers. Each has one job, a defined interface, and can be tested alone.

```
web/          checkout simulator + ops dashboard
  │ HTTP
api/          FastAPI — /promise, /lanes, /sellers, /investigation
  │
agent/        Nemotron function-calling loop — orchestrates, narrates
  │ tools
analysis/     deterministic Python — owns ALL math and the verdict
  │
craft/        MCP client (OAuth PKCE) + cassette record/replay
  │
CRAFT MCP → Olist (Snowflake, read-only)
```

**The load-bearing invariant: numbers never pass through the LLM.** The LLM chooses which
questions to ask and writes prose. Every figure it states is one the analysis layer computed and
handed to it. This is what makes the output trustworthy under a judge's "why 38 days?"

### 2.1 `craft/` — data access

- `client.py` — MCP client over Streamable HTTP + OAuth 2.1 (Authorization Code + PKCE) against
  `https://nebius.emergence.ai/mcp`, with `X-Project-ID` header.
- `cassette.py` — records every CRAFT request/response pair to `fixtures/<hash>.json`. On replay,
  a request that hashes to a known fixture returns the recorded response without network access.
- `questions.py` — the natural-language questions, verbatim from the spec. **No hand-written
  SQL**: all SQL comes from `generate_sql`.

Three modes, selected by env:
| Mode | Behavior |
| --- | --- |
| `replay` (default) | Fixtures only. No network, no credentials. Errors on a cache miss. |
| `record` | Live CRAFT; writes fixtures as it goes. |
| `live` | Live CRAFT; no fixture writes. |

**Known CRAFT gotchas to encode in the client:**
- `execute_query` does **not** return rows — it returns an `artifact_fqn`; you must then call
  `get_result_page`.
- `sample_data` wants a 3-part name (`DB.SCHEMA.TABLE`), not the 4-part FQN.
- Olist dates are `VARCHAR` → `TRY_TO_TIMESTAMP`.
- Rate limits: metadata 30/min, query execution 10/min.

Because `generate_sql` is non-deterministic, fixtures record the **generated SQL and the result
rows**, and tests assert on the shape of results — never on SQL text. Otherwise the suite breaks
whenever CRAFT's model updates.

### 2.2 `analysis/` — the deterministic core

```
promise(seller, state) = handling_p95(seller) + transit_p95(state)

# seasonality is opt-in:
promise(seller, state, month, seasonal=True)
    = handling_p95(seller) + transit_p95(state) × season_factor(month)
```

- `handling_p95(seller)` — `order_approved_at → order_delivered_carrier_date`, 95th percentile.
  Observed range across sellers: 2.1 → 5.0 days.
- `transit_p95(state)` — `order_delivered_carrier_date → order_delivered_customer_date`, 95th
  percentile, by `customer_state`.
- `season_factor(month)` — derived from the monthly late-rate table, which swings 1.4% → 21.4%.
  Normalized so a median month is 1.0.

**Seasonality is a flag, off by default** (`seasonal=False`, `?seasonal=true` on the API). The
base promise is a pure two-term decomposition — that's the claim we defend. The season factor is
fit on only ~2 years of data and one Black Friday, so it is offered as an explicit adjustment the
caller opts into, and when enabled it appears as its own line in the attribution rather than
being silently folded into the total. This keeps the core number defensible and makes the Black
Friday story a demonstrable toggle rather than an assumption baked into every quote.

Sellers or lanes below a support threshold (50 delivered items / 500 orders) fall back to the
national aggregate rather than reporting a noisy percentile.

### 2.3 The pad/fix verdict

The gap decomposes into two components:

- **distance component** = `median_transit` — irreducible. The lane really is far.
- **variance component** = `p95_transit − median_transit` — recoverable. This is the tail.

If the variance component dominates the promise, the lane is *unpredictable, not slow* → **FIX**.
If the median itself is what's large, the lane is genuinely far → **PAD**.

Rio: median 12, p95 38 → 26 of the 38 days are tail → **FIX**.
Pará: high median → **PAD**. Padding is honest; it really is far away.

The same rule produces opposite verdicts with no per-lane hand-tuning. The exact threshold is
calibrated against the data during implementation and committed as a named constant with its
justification, not scattered as a magic number.

A lane whose gap is within tolerance of zero (SP, MG, PR) returns **OK** — no action. The engine
must not cry wolf on already-calibrated lanes; that restraint is part of the product.

### 2.4 `agent/` — the investigation loop

Nebius Token Factory (`nvidia/nemotron-3-super-120b-a12b`), OpenAI-compatible, native function
calling. Provider-agnostic client: any OpenAI-compatible `base_url` works.

Tools exposed to the LLM:

| Tool | Returns |
| --- | --- |
| `ask_craft(nl_question)` | Rows from CRAFT via `generate_sql` → `execute_query` → `get_result_page` |
| `test_hypothesis(name)` | A structured verdict: SURVIVES / DEAD, with the evidence |
| `compute_promise(seller, state, month)` | Promise, attribution, verdict — from `analysis/` |
| `rank_lanes()` | Ops queue: lanes by gap × volume |

The agent runs the falsification suite **before** recommending: churn (dead — 3.12% repeat, flat
by review score), bad-sellers (dead — volume artifact; no seller is even 40% late), flat-buffer
(dead — it *is* distance-adjusted). Only then does it recommend. This is the architecture score:
it investigates and discards, rather than shipping the first chart it found.

When the LLM has no key configured, the agent falls back to a scripted investigation path over
the same tools, so the demo always runs.

### 2.5 `api/` + `web/`

FastAPI:
- `POST /promise` — `{seller_id, customer_state, month, seasonal=false}` → promise, attribution,
  verdict, narrative. With `seasonal=true`, the response carries an extra attribution line showing
  what the season factor added or removed.
- `GET /lanes` — the ops work-queue.
- `GET /sellers` — handling-time scorecards.
- `GET /investigation` — the falsification preamble, as structured data.

Web app, two views:
1. **Checkout** — pick a seller and a destination; see the promise Olist *would* show, the promise
   the engine recommends, and *why* ("5 days of that is this seller's handoff; 26 days is Rio's
   tail").
2. **Ops** — the ranked work-queue with the pad/fix verdict per lane.

## 3. Testing

TDD. The analysis layer is written test-first against the verified ground truth above.

- **Regression tests on known truth.** Rio → promise 38, gap +11, verdict FIX. SP/MG/PR → gap ≈ 0,
  verdict OK. Pará → verdict PAD. These are the numbers already confirmed against live data;
  if a refactor breaks them, the suite fails.
- **Verdict rule unit tests.** Synthetic lanes at the boundary — a high-median/low-tail lane must
  PAD, a low-median/high-tail lane must FIX, at every support level.
- **Cassette contract test.** A recorded session replays byte-identically with no network.
- **The invariant test.** The agent's narrative may not contain a figure the analysis layer did not
  produce. Numbers in the LLM's output are extracted and checked against the computed set.
- **End-to-end.** The web app is driven for real before the work is called done.

Tests run offline, on fixtures, with no credentials.

---

## 4. Risks

| Risk | Mitigation |
| --- | --- |
| CRAFT OAuth PKCE from a standalone process is fiddly | Replay mode is the default; the build never blocks on auth |
| `generate_sql` returns different SQL run-to-run | Fixtures assert on result shape, never SQL text |
| p95 on a thin seller is noise | Support thresholds, with fallback to national aggregate |
| Season factor overfits 2 years of data | Opt-in flag, off by default; when on, it is a bounded multiplier reported as its own attribution line |
| LLM states a number it invented | The invariant test fails the build |
