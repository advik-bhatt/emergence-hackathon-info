Why deps-dev-v1

Three realistic non-ecommerce / non-biotech options. Two lose — and it comes down to a single column nobody else has.

crypto Ground-truth gap

Ethereum: 18.5M token transfers (2016→2024). Viable, but a named challenge — so not uncontested — and fraud findings have no ground truth. Demo reduces to “these wallets look suspicious, trust me.”

ga4 / firebase Weak story

The hardest SQL in the set (nested LATERAL FLATTEN), which flatters usage depth — but the deliverable is a funnel dashboard. Nothing a judge repeats afterward.

deps-dev-v1 ✓ Winner

The reason is one column nothing else has: DEPENDENTS.MinimumDepth. The transitive dependency graph is precomputed — so blast radius, the thing real security tooling struggles to compute, becomes a join instead of a graph traversal.

Landmines

Verified the hard way. Read these before writing a line of SQL.

▲

deps-dev-v1 is NOT a time series of the graph. ADVISORIES and DEPENDENTS have exactly one snapshot each. Only PACKAGEVERSIONS (44) and DEPENDENCIES (19) have history. Any “watch a vuln spread over snapshots” plan dies on contact.

✦

The real time signal is event timestamps — and it's better. ADVISORIES.Disclosed and PACKAGEVERSIONS.UpstreamPublishedAt are true event times: “this CVE has been public for 2,100 days and N packages still ship a vulnerable version.” No snapshot history required.

▲

sample_data takes a 3-part name (DB.SCHEMA.TABLE), not the 4-part FQN the docs advertise. It will 400 on you otherwise.

▲

All timestamps are microsecond epoch integers. Divide by 1,000,000 before TO_TIMESTAMP.

▲

github-repos is a sample, not all of GitHub. No issues / PRs tables. Don't plan around contributor history.
