# Project Ideas — Emergence × Nebius (CRAFT)

Every claim below was **verified by querying the data through the CRAFT MCP**, not read off
the docs. Several things the docs assert turned out to be false — see Landmines.

**The only hard requirement is: build on at least one database.** No mandatory challenge
track, no forced tech. So the design goal is simply to maximise the rubric:

**CRAFT usage depth 30% · Insight quality 30% · Agent architecture 20% · Story clarity 20%**

Two consequences worth internalising:

- **60% of the score is insight + CRAFT depth.** Pick the dataset that has a *real finding*
  in it. Everything else is decoration.
- Spanning **multiple connections** is the cheapest way to prove "CRAFT usage depth."
- Nebius is *strategic, not mandatory*: the judges are Emergence + Nebius engineers and the
  prizes are Nebius credits. Use Nemotron-3 Super 120B as the reasoning brain because it's
  smart politics and it's genuinely good at tool use — not because a rule forces you.

## ⚠️ Read this before choosing a dataset: `thelook` is synthetic

`thelook-ecommerce` is the dataset the reference agent used, so it is where the crowd will
pile in. **It has no latent structure.** Asked the classic CFO question — *do some
acquisition channels bring customers who return so much they're unprofitable?* — every
channel came back identical:

| Channel | Return rate | Net margin / customer |
| --- | --- | --- |
| Organic | 11.1% | $57.79 |
| Email | 12.1% | $58.87 |
| Display / Facebook / Search | 11–12% | ~$58 |

That is a random number generator, not a business. `thelook` was generated by sampling
independent distributions, so **any correlation you go hunting for will come back null.**
With *Insight Quality* at 30%, this dataset is close to unwinnable — you can write beautiful
SQL and still have nothing to say.

**Contrast with `brazilian-e-commerce` (Olist), which is real data:**

| Delivery vs. promise | Orders | Avg review | % 1-star |
| --- | --- | --- | --- |
| Early | 88,653 | 4.29 | 6.6% |
| On time | 1,291 | 4.03 | 8.5% |
| 1–3 days late | 1,856 | 3.29 | 25% |
| 4–7 days late | 1,756 | 2.10 | 58.5% |
| 8–15 days late | 1,609 | 1.68 | **70%** |

Clean, monotonic, brutal. *That* is what a real signal looks like.
**If you touch ecommerce at all, use Brazilian — never `thelook`.**

## Dataset selection: why `deps-dev-v1`

Three realistic non-ecommerce/non-biotech options, and why the other two lose:

- **`crypto`** — viable (Ethereum: 18.5M token transfers, 2016-04 → 2024-07), but it's a
  *named challenge*, so it is not actually uncontested. Worse: fraud/structuring findings
  have **no ground truth**. The demo reduces to "these wallets look suspicious, trust me,"
  which judges cannot verify. Scores badly on *Insight quality*.
- **`ga4` / `firebase`** — the hardest SQL in the set (nested `LATERAL FLATTEN`), which
  flatters *CRAFT usage depth*, but the deliverable is a funnel dashboard. Weak story.
- **`deps-dev-v1`** — ✅ **winner.**

The reason is one column nothing else has: **`DEPENDENTS.MinimumDepth`**. The transitive
dependency graph is *precomputed*. Blast radius — the thing real security tooling
struggles to compute — becomes a join instead of a graph traversal. Verified scale:

| Ecosystem | Dependent edges | Distinct packages | Distinct dependents | Max depth |
| --- | --- | --- | --- | --- |
| NPM | ~535,000 | 32,675 | 155,336 | **117** |

MAVEN, GO, PYPI, CARGO follow.

## 💡 Idea 1 (recommended): **Blast Radius** — a dependency upgrade-triage agent

### The insight nobody else will have

Every SBOM scanner on earth tells you *"you have 47 vulnerabilities."*
**None of them tell you "those 47 collapse into 3 upgrades."**

That is a **minimum set-cover problem over the dependency graph**, and `deps-dev-v1` is
the only dataset here that can express it. That single reframing is the whole pitch:
we don't rank vulnerabilities, we rank *actions*.

### What it does

1. User pastes a `package.json` (or picks a real repo — see validation corpus below).
2. Agent resolves the transitive dependency tree via `DEPENDENCIES` / `DEPENDENTS`.
3. Joins to `ADVISORIES` to find CVEs affecting resolved versions.
4. Scores each vuln by **reach** (how many of *your* packages it sits under) and
   **depth** (`MinimumDepth` — a CVE 6 hops down behind a dev-only path is not your
   Tuesday problem; one at depth 1 is).
5. **Solves for the smallest set of upgrades that removes the most risk** — and prints
   it as an ordered action list, not a vulnerability table.

### Verified supporting data

The core join works. NPM packages with a CRITICAL advisory (CVSS ≥ 9.0), ranked by
distinct downstream dependents:

| Package | Advisory | CVSS | Disclosed | Downstream dependents |
| --- | --- | --- | --- | --- |
| lodash | Prototype Pollution | 9.1 | 2019-07-10 | **1,254** |
| minimist | Prototype Pollution | 9.8 | 2022-03-18 | 871 |
| https-proxy-agent | Denial of Service | 9.1 | 2018-07-27 | 358 |
| loader-utils | Prototype pollution | 9.8 | 2022-10-13 | 314 |
| json-schema | Prototype Pollution | 9.8 | 2021-11-19 | 304 |
| uglify-js | Incorrect Handling of Non-Boolean Comparisons | 9.8 | 2017-10-24 | 229 |

### The line that wins the room: **load-bearing but abandoned**

Fold in a second risk signal — packages a huge chunk of the ecosystem depends on, whose
last release is ancient. Verified:

- **`once`** — **941** packages depend on it. Last published **September 2016**.
- **`isarray`** — **1,239** dependents. Last published July 2019.
- **`lodash`** — **1,254** dependents, *and* an unfixed-in-the-wild critical CVE from 2019.

> *"941 packages depend on code nobody has touched since 2016."*

That is the demo line judges repeat to each other afterwards. It is also a genuinely
novel metric — a "left-pad risk" score — not a restatement of what Dependabot says.

### The cross-connection flex

`github-repos.SAMPLE_CONTENTS` contains **real manifest files**:

| Manifest | Rows | Distinct repos |
| --- | --- | --- |
| `package.json` | 80 | 76 |
| `pom.xml` | 45 | 44 |
| `requirements.txt` | 4 | 4 |
| `Cargo.toml` | 3 | 3 |
| `go.mod` | 0 | 0 |

Too thin to be a product, **perfect as a validation slide**: *"we ran this against 76 real
GitHub repos — here's the leaderboard of most-exposed projects."* It's a genuine
**two-connection join** (`github-repos` → `deps-dev-v1`), the same structural difficulty
as the biotech track's TCGA join, on a dataset nobody is fighting over.

### Why it scores on the rubric

- **CRAFT usage depth (30%)** — `LATERAL FLATTEN` over the `Packages`, `Advisories`,
  `Licenses`, and `Dependent` VARIANT columns; a two-connection join; all SQL via
  `generate_sql` (no hand-written SQL anywhere).
- **Insight quality (30%)** — set-cover over upgrades + the abandonment score are real,
  verifiable, non-obvious findings.
- **Agent architecture (20%)** — natural multi-tool loop: resolve tree → find CVEs →
  score reach/depth → optimize upgrade set → explain.
- **Story clarity (20%)** — "47 vulns → 3 upgrades" and "941 packages, untouched since
  2016." Both land in one sentence.

### Demo

Force-directed graph of the dependency tree, vulnerable nodes glowing red. Agent
recommends 3 upgrades. Apply them → **the red collapses.** Visual, instant, legible from
the back of a room.

## 🎙️ The voice layer: "the 2am page"

A teammate has effectively unlimited **ElevenLabs** credits. A voice agent is the one thing
nobody at a *text-to-SQL* hackathon will have. But be clear-eyed about why:

**Voice earns almost nothing directly.** There is no UI/UX category in the rubric. Voice
touches only the 20% *Story clarity* slice, while 60% of the score is insight + CRAFT depth,
neither of which a voice layer improves. The trap is spending four hours on something that
demos for ninety seconds and leaves the actual finding thin. In a noisy breakout room, a
live voice call can also simply fail.

**Do it anyway — but only because this specific framing makes voice the *correct* modality,
not a gimmick.**

Dependency incident response is **already a phone-first workflow.** It's 2am, you're paged
that a critical CVE dropped. You are not opening a BI dashboard. You're on your phone, half
awake, and you need exactly three things: *how bad is it, what do I do, can it wait?*

So: **the phone rings on stage.** You pick up and ask "how bad is it?"

> *"Critical. Prototype pollution in lodash, CVSS 9.1. It reaches 1,254 packages downstream
> of you, but it's four hops deep behind a dev dependency. Three upgrades clear 80% of your
> exposure — start with minimist. This can wait until morning."*

Hands-busy user, urgent context, and the answer is a **decision, not a table**. That's the
test a voice interface has to pass, and this passes it. It also sells the exact same
set-cover insight that scores on the 30% axis — voice is the delivery, not the substance.

### Architecture — keep Nemotron as the brain

```
Phone / mic
   │
ElevenLabs  ── STT + TTS ONLY (voice in, voice out)
   │
Your agent loop
   │
Nebius Token Factory — Nemotron-3 Super 120B  ← reasoning + function calling
   │
CRAFT MCP — generate_sql → execute_query → get_result_page
```

**Do not let ElevenLabs' built-in agent LLM do the thinking.** Nemotron-3 Super 120B has
native function calling and 1M context; it should own the tool loop. Use ElevenLabs purely
as ears and mouth. (If you want ElevenLabs Agents for turn-taking / interruption handling,
run it in **custom-LLM mode** pointed at Nebius's OpenAI-compatible endpoint — same outcome,
Nemotron still reasoning.)

### Build sequence — voice is strictly last

1. **The investigation.** Nail the set-cover finding + abandoned-package score. This is 60%.
2. **The agent loop** — real multi-step reasoning and error recovery. This is the 20%
   architecture score, and where CRAFT's DX tools (`search_schema`, `explain_error`,
   `get_hint`) earn depth points.
3. **Voice, time-boxed.** The text chat path stays fully working underneath.
4. **Record a backup demo video the moment voice works.** If the room's WiFi or mic betrays
   you, play the tape and lose nothing.

> **The rule:** if voice isn't working by your cutoff, ship without it and the project is
> still strong. If that's not true, the layering is wrong.

## 💡 Idea 2: **Cohort Bias Auditor** (`idc` × `pancancer-atlas-1`) — best health idea

Every medical AI model is trained on the patients who *happened to get scanned*. Are those
patients representative of the actual disease population? Almost nobody checks.

**Verified:** for breast cancer, IDC has imaging for **562** TCGA patients, while the
PanCancer clinical table has **1,087**. Roughly **half the population is invisible** to any
imaging-trained model. The join is real — IDC `PatientID` is literally `TCGA-BH-A0BV`, the
same format as the clinical `bcr_patient_barcode`. 32 TCGA imaging collections exist.

The clinical table (10,761 patients, 32 cancer types) carries `race`, `ethnicity`, `gender`,
`age_at_initial_pathologic_diagnosis`, `pathologic_stage`, `vital_status`, `days_to_death`.
So you can ask **who is missing**.

Output: *"Your imaging cohort under-represents Black patients by X% and over-represents
Stage II. A model trained here will underperform on exactly the groups with worse outcomes."*

- **Real** — FDA and every health system cares about exactly this.
- **Unique** — everyone else on the biotech track will do "mutation X → survival."
- It is the **genuine two-connection join**, which is where CRAFT depth points live.
- Demos as one devastating bar chart.

## 💡 Idea 3: **Trial Feasibility Agent** (`pancancer-atlas-1`)

Speak inclusion criteria in plain English — *"Stage III breast cancer, treatment-naive,
under 60"* — and the agent reports how many of the 10,761 real patients qualify, and **which
criterion is killing your enrollment**. ~80% of clinical trials miss enrollment targets
because criteria are written blind. Loosen one criterion, watch eligible N jump.

Strongly agentic, real use case, and it pairs *extremely* well with the voice layer (a trial
coordinator on the phone is a natural fit).

## 💡 Idea 4: **Delivery Promise Optimizer** (`brazilian-e-commerce`)

*Not* "late delivery causes bad reviews" — that's the reference agent, and it's a
**diagnosis**. Instead treat the estimated delivery date as a **decision variable**.

Look again at the table above: **88,653 orders arrive early, only 1,291 on time.** Olist is
sandbagging its delivery estimates enormously. So: per region and per seller, find the most
*aggressive* promise date that still lands ~95% on-time. Faster promises convert better, and
you're currently paying for padding you don't need.

That's a **recommendation, not an observation** — which is what "actionable insight" actually
means on the rubric.

## 💡 Idea 5: Ecosystem Fragility Index

Score the "left-pad risk" of an entire ecosystem: high dependent count × dead project
(`PROJECTS` stars / open issues) × stale `UpstreamPublishedAt`. Output is a ranked list of
open source's single points of failure.

Genuinely novel and arguably publishable — **but it is an *analysis*, not an agent.** It
demos as a chart. Weak fit for an agent hackathon (kills the 20% architecture score).

**Better used as a feature inside Idea 1**, which is exactly where it's most actionable
anyway.

## 💡 Idea 6: Crypto circular-flow detection

Visually stunning graph.

## Landmines (verified the hard way)

- **`thelook` is synthetic and has no latent signal.** See the top of this file. Any
  correlation you hunt for returns null. It is also the most crowded dataset. Avoid.
- **Don't let voice eat the investigation.** 60% of the score is insight + CRAFT depth;
  voice touches only the 20% story slice. Build it last, time-boxed, with a text fallback
  and a recorded backup demo.
- **`deps-dev-v1` is NOT a time series of the dependency graph.** `ADVISORIES` and
  `DEPENDENTS` have exactly **one snapshot each**. Only `PACKAGEVERSIONS` (44 snapshots)
  and `DEPENDENCIES` (19) have history. Any plan that assumes you can "watch a vuln
  spread over time" via snapshots will die on contact.
- **The real time signal is event timestamps, and it's better anyway:**
  `ADVISORIES.Disclosed` and `PACKAGEVERSIONS.UpstreamPublishedAt` are true event times.
  That's how you say *"this CVE has been public for 2,100 days and N packages still
  depend on a vulnerable version"* — no snapshot history required.
- **`sample_data` takes a 3-part name** (`DB.SCHEMA.TABLE`), not the 4-part FQN the tool
  docs advertise. It will 400 on you otherwise.
- **All timestamps are microsecond epoch integers.** Divide by 1,000,000 before
  `TO_TIMESTAMP`.
- **`github-repos` is a sample, not all of GitHub.** No issues/PRs tables. Don't plan
  around contributor history.
